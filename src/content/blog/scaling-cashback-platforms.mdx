## The Spike Problem Nobody Warns You About

Campaign products behave very differently from ordinary e-commerce. A normal Tuesday can become your worst traffic event of the year in under ten minutes when a flash sale goes live. I've been through this enough times at BigTricks and the broader cashback platform ecosystem to know: the systems that survive are built for the spike, not the average.

The first time we saw a 40x traffic surge during a major bank cashback campaign, the bottleneck wasn't the server — it was the transaction-tracking pipeline. Every user click was triggering a synchronous write to the database, a call to the merchant tracking API, and a cashback eligibility check before the redirect fired. At scale, that chain breaks fast.

## Make Everything Measurable First

The first improvement is always observability. You cannot optimize what you cannot measure, and campaign platforms have an unusually complex funnel: impression → click → redirect → merchant conversion → cashback credit. Each step can fail independently, and each failure mode looks different.

We instrumented every stage with structured logs and custom metrics — click-to-redirect latency, merchant API response time distribution, cashback calculation queue depth, and credit posting lag. Once the full funnel was measurable, the bottlenecks became obvious within hours rather than days of investigation.

Practical starting point: emit a single structured JSON log per transaction event with a shared `session_id` that flows through every service. You can reconstruct the full journey in CloudWatch Logs Insights or Grafana without a distributed tracing setup.

## Caching the Right Things

Not all data needs to be fresh. Merchant metadata — offer terms, cashback rates, eligible categories — changes at most a few times per day. Serving that from cache eliminates dozens of database reads per second during peak periods.

We implemented a two-level cache: an in-memory LRU cache per instance for the hottest merchants, with Redis as the second level for anything that fell out of memory. TTLs are short enough that rate changes take effect within minutes, but long enough to absorb traffic spikes without the database ever being the bottleneck for reads.

The rule we follow: if the data can be stale for 60 seconds without breaking a user's experience, cache it aggressively.

## Queue-Based Processing for Anything Non-Critical

Cashback credits don't need to post in real time. The user just needs to see their click tracked and trust that the credit will appear. Moving credit calculation and posting to an async queue (we use SQS with dead-letter queues for failures) removed an entire class of latency from the hot path and gave us natural retry capability when merchant APIs had intermittent outages.

The redirect itself became a thin, fast operation: validate the click, write a lightweight event to the queue, fire the redirect. Everything else is asynchronous and can scale independently.

## Resilient Retries, Not Blind Retries

One anti-pattern I see often: retry logic that makes slow systems slower. If a merchant API is timing out, retrying immediately just adds load. Exponential backoff with jitter, a per-merchant circuit breaker, and a fallback to the last known-good cashback rate gave us resilience without the thundering herd problem.

The results were concrete: during a large campaign event, we brought P99 redirect latency from 1,800ms to under 200ms, reduced merchant API error rate from 12% to under 1%, and cut infrastructure costs by roughly 30% by eliminating redundant synchronous database calls.

The pattern is repeatable: measure everything, cache what's safe to cache, make everything outside the hot path asynchronous, and build retry logic that respects the systems it depends on.
